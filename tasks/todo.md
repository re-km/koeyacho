# Workflow Plan: Gemini Integration into GAS Backend

## 1. 調査・分析
- [x] `index.html` で音声認識の生テキスト（rawText）がバックエンドに送信されているか確認する。
- [x] 送信されていない場合は、`index.html` を修正し、生テキストを保存・送信するようにする。
- [x] `gas_backend.js` に Gemini API を組み込むための基本構成（PropertiesService の利用など）を検討する。

## 2. 実装: フロントエンド (`index.html`)
- [/] 音声認識結果の生テキストを送信データに含める。
- [/] クイック入力ボタン、手動入力エリア、辞書登録モーダルなど不要な入力UIを削除する。
- [/] UIに必要な要素を残す: **選択中の橋梁名**、**入力先のシート名**、**自動採番（入力件数）**。
- [/] **スプレッドシートを開くボタン**を追加し、現在選択中の橋梁のファイルURLに飛べるようにする。
- [/] `koeyacho_dictionary.js` の読み込みや関連処理を削除する。
- [/] **入力履歴機能**: 入力した内容（生テキストの羅列）をリストとして画面下部に表示する。
- [/] **編集機能**: リストの各項目に「編集」ボタンを設け、記録された生テキストを直接テキストボックスで修正・再送信できるようにする。
- [/] **設定のハードコード化**: GASのWeb App URLを `index.html` 内の変数に直接書き込む（ハードコードする）ことで、LocalStorageクリアによる消失を防ぐ。

## 3. 実装: バックエンド (`gas_backend.js`)
- [x] Gemini API 連携用の関数 (`callGeminiAPI`) を追加する。モデルは `gemini-2.5-flash` を利用する。
- [x] `system_prompt.md` の内容を踏襲し、生テキストをGeminiに渡してJSON形式でパース結果を受け取る処理を実装する。
- [x] `doPost` 関数内で Gemini を呼び出し、結果を用いてスプレッドシートへの書き込みデータを生成する。
- [x] 古いパラメータの受付を削減し、`data.rawText` を主軸とした処理に変更する。

## 4. 検証
- [ ] Gemini API キーをGASのプロパティに設定し、テスト実行する。
- [ ] フロントエンドからダミーの誤変換テキストを送信し、正しく翻訳されてスプレッドシートに記録されるか確認する。
